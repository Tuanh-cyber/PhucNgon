// 01

Tổng Quan Kiến Trúc Scoring Engine

Scoring Engine nhận vào transcript từ ASR và metadata của bài tập, trả về điểm số và kết quả phân loại. Engine được chia thành 4 sub-scorer riêng biệt tương ứng với 4 sub-mode của 3 exercise type. Mỗi sub-scorer có công thức trọng số khác nhau phản ánh yêu cầu lâm sàng của từng loại bài.

pythonscoring_engine/router.py
def score(exercise: Exercise, transcript: str, audio_duration: float) -> ScoreResult:
    # Gate 1: reject invalid ASR output before scoring
    if is_invalid_input(transcript, audio_duration):
        return ScoreResult.invalid()

    # Route to correct sub-scorer by exercise type + mode
    match exercise.type:
        case "naming":
            return score_naming(exercise, transcript, audio_duration)

        case "command_identification":
            if exercise.mode == "recognition":
                return evaluate_cmd_recognition(exercise, selected_vocab_id)  # Correct/Incorrect, no score
            else:
                return score_cmd_repetition(exercise, transcript, audio_duration)

        case "sentence_building":
            return score_sentence(exercise, transcript, audio_duration, attempt_number)
NAM
Naming
Nhìn ảnh → nói tên. Trọng tâm: keyword match + classifier. Dùng ASR.
CMD / Mode 1
Recognition
Nghe mô tả → chọn đáp án touch. Chỉ Correct / Incorrect, không tính điểm. Không dùng ASR.
CMD / Mode 2
Repetition
Nghe mô tả + nhìn ảnh → nói tên. Như Naming nhưng đã có audio cue.
SEN
Sentence Building
Câu khuyết + ảnh → nói đủ câu. Phức tạp nhất: 5 thành phần scoring.

Input gate: Trước khi gọi bất kỳ sub-scorer nào, hệ thống kiểm tra: audio_duration < 0.5s → reject; transcript == "" → retry; asr_confidence < 0.4 → retry. Các case này không đưa vào scoring và không tính vào attempt_count.

// 02

Text Normalization Pipeline

Áp dụng cho mọi sub-scorer sử dụng ASR (NAM, CMD Mode 2, SEN). CMD Mode 1 không cần normalize vì không dùng ASR.

pythonscoring_engine/normalize.py
import unicodedata
import re

def normalize(text: str) -> str:
    # 1. Unicode NFC — chuẩn hóa dấu tiếng Việt (precomposed form)
    text = unicodedata.normalize("NFC", text)

    # 2. Lowercase
    text = text.lower()

    # 3. Strip punctuation: xóa .,!?… nhưng GIỮ dấu tiếng Việt
    text = re.sub(r"[^\w\s]", "", text, flags=re.UNICODE)

    # 4. Collapse whitespace
    text = re.sub(r"\s+", " ", text).strip()

    return text

// 03

NAM — Naming Score

Bệnh nhân nhìn ảnh → nói tên đồ vật. Trọng tâm là từ khóa chính có xuất hiện không, và có kèm loại từ đúng không. Target so sánh là canonical_word trong Vocabulary Asset.

// naming_score formula
score = 0.50 × keyword_match + 0.25 × text_similarity + 0.15 × fluency + 0.10 × classifier_present
keyword_match · 50%
100 nếu transcript chứa bất kỳ answer nào trong accepted_answers. 0 nếu không có.
text_similarity · 25%
Levenshtein ratio giữa transcript normalize và canonical_word normalize × 100.
fluency · 15%
Duration ratio bệnh nhân / TTS. Xem Section 07.
classifier_present · 10%
100 nếu transcript chứa loại từ nằm trong accepted_classifiers của chính vocab đó. 0 nếu không có hoặc dùng sai loại từ.
Trọng số keyword_match = 50% vì Naming đo khả năng gọi đúng tên — đây là mục tiêu lâm sàng chính của bài. Text_similarity hỗ trợ credit partial khi bệnh nhân gần đúng âm.

Classifier Lookup — dùng accepted_classifiers

Classifier check dựa trên field accepted_classifiers được định nghĩa riêng cho từng vocab trong Asset. Chỉ chấp nhận loại từ nằm trong danh sách classifier đúng của vocab đó — loại từ tồn tại nhưng sai ngữ nghĩa (vd "tờ" cho "kéo") không được tính.

python
# accepted_classifiers lưu trong Asset.xlsx → Vocabulary
# Ví dụ V001 "cái kéo" → accepted_classifiers = "cái|cây"

def check_classifier(transcript: str, accepted_classifiers: list[str]) -> float:
    if not accepted_classifiers:
        return 100.0  # vocab không yêu cầu classifier (verb/adj) → không phạt
    words = transcript.split()
    # Chỉ chấp nhận classifier ĐÚNG với vocab này
    return 100.0 if any(c in words for c in accepted_classifiers) else 0.0

// 04

CMD Mode 1 — Recognition (Correct / Incorrect)

Bệnh nhân nghe mô tả âm thanh → chọn đáp án đúng bằng touch (4 options). Hoàn toàn không dùng ASR. Đây là bài tập kiểm tra khả năng thông hiểu thính giác, không đo phát âm.

// cmd_recognition — KHÔNG tính điểm số
result = CORRECT  if  selected_vocab_id == target_vocab_id  else  INCORRECT
CMD Mode 1 KHÔNG sinh score 0–100. Bài touch chỉ có 2 trạng thái: Correct / Incorrect. Không có partial credit, không có Near. CORRECT được tính như một "bài đúng" trong chuỗi progression (xem Section 09); INCORRECT reset chuỗi.

Tại sao không tính điểm? CMD Recognition bản chất là bài trắc nghiệm thông hiểu — không có gì để chấm điểm từng phần. Việc ép nó vào thang 0–100 là gượng ép. Hiển thị Correct/Incorrect trung thực hơn với bản chất bài, và vẫn tham gia bình thường vào progression rule (đếm số bài đúng liên tiếp).

Input & Output của CMD Mode 1

pythonscoring_engine/cmd_recognition.py
def evaluate_cmd_recognition(
    exercise: Exercise,
    selected_vocab_id: str  # ID vocab bệnh nhân tap chọn, từ frontend
) -> RecognitionResult:

    is_correct = (selected_vocab_id == exercise.target_vocab_id)

    return RecognitionResult(
        result      = "correct" if is_correct else "incorrect",
        score       = None,   # KHÔNG có điểm số
        selected_id = selected_vocab_id,
        is_correct  = is_correct,
        # Tham gia progression: correct → +1 streak, incorrect → reset
        counts_for_progression = True,
    )

// 05

CMD Mode 2 — Repetition Score

Bệnh nhân nghe mô tả + nhìn ảnh target → nói tên từ. Giống Naming nhưng bệnh nhân đã có 2 cue đồng thời (nghe audio mô tả + thấy ảnh) trước khi nói. Vì đã nghe đáp án nên khả năng cao bệnh nhân sẽ nói đúng từ khóa — do đó điểm phân biệt thực sự nằm ở sự hoàn thiện của đáp án (text_similarity + classifier), không phải ở keyword.

// cmd_repetition_score formula
score = 0.40 × keyword_match + 0.30 × text_similarity + 0.20 × classifier_present + 0.10 × fluency
keyword_match · 40%
Giảm so với NAM — vì đã nghe cue nên nói đúng keyword gần như là baseline, không còn là điểm phân biệt chính.
text_similarity · 30%
Tăng — đánh vào độ hoàn thiện về âm/từ của đáp án sau khi đã nghe mẫu.
classifier_present · 20%
Cao hơn NAM (10%) — đã có đủ cue mà vẫn thiếu loại từ là lỗi đáng chú ý hơn. Dùng accepted_classifiers như NAM.
fluency · 10%
Duration ratio. Giảm xuống 10% vì bài ngắn (1–2 từ), ít có ý nghĩa phân biệt.
Triết lý: đã nghe đáp án → keyword là điều kiện gần như mặc định. Điểm phân biệt chuyển sang text_similarity + classifier (tổng 50%) để đo mức độ hoàn thiện của đáp án sau khi được hỗ trợ.

SEN — Sentence Building Score

Bệnh nhân nhìn câu khuyết + ảnh → nói đủ câu hoàn chỉnh. Ví dụ: câu "Tôi đang ___" + ảnh mô tả hành động ăn cơm → bệnh nhân phải nói được "tôi đang ăn cơm". Mục tiêu tối thiểu: nhận diện được hành động và nói ra từ khuyết, kèm cấu trúc câu phù hợp. Target là full_sentence trong Sentence Instance Asset.

// sentence_score formula (attempt 1 — chưa nghe fallback audio)
score = 0.40 × keyword + 0.50 × order_score + 0.10 × fluency
keyword · 40%
Phần từ khuyết (vd "ăn cơm") có được nói đủ không. Đo mục tiêu chính: nhận diện hành động.
order_score · 50%
Thứ tự từ toàn câu. Đo cấu trúc câu — điểm phân biệt chính của bài "xây câu" so với "gọi tên".
fluency · 10%
Duration ratio. Giữ thấp — không phạt nặng bệnh nhân nói chậm (đặc trưng aphasia).
order_score = 50% (nặng nhất) vì cấu trúc câu là mục tiêu cốt lõi. keyword = 40% nhấn mạnh việc nhận ra từ khuyết. Hai thành phần này phối hợp: keyword đo "đủ từ khuyết", order đo "đúng thứ tự toàn câu".

keyword — Phần từ khuyết

Phần từ khuyết được lưu trong field riêng của Sentence Instance (do người làm exercise bank bổ sung — lưu chính xác phần bị che trong câu). keyword chỉ đo có nói đủ từ khuyết không, không xét thứ tự (thứ tự để order_score lo).

python
from underthesea import word_tokenize

def keyword_score(transcript: str, missing_words: list[str]) -> float:
    # missing_words = phần khuyết trong câu, vd ["ăn", "cơm"]
    if not missing_words:
        return 100.0
    trans_tokens = set(word_tokenize(normalize(transcript)))
    matched = [w for w in missing_words if w in trans_tokens]
    return len(matched) / len(missing_words) * 100

order_score — Thuật toán thứ tự từ

Duyệt từng từ của target theo thứ tự, tìm vị trí đầu tiên của nó trong câu bệnh nhân. Phân loại mỗi từ target thành: đúng vị trí / sai vị trí / thiếu. Từ target đầu tiên luôn là mốc (đúng vị trí).

python
def order_score(transcript: str, target: str) -> float:
    target_seq = word_tokenize(normalize(target))
    trans_seq  = word_tokenize(normalize(transcript))

    correct, misplaced, missing = 0, 0, 0
    last_pos = -1  # vị trí của từ target trước đó trong câu bệnh nhân

    for word in target_seq:
        # tìm vị trí ĐẦU TIÊN của word trong câu bệnh nhân
        pos = trans_seq.index(word) if word in trans_seq else None

        if pos is None:
            missing += 1                     # không tìm thấy → thiếu
        elif last_pos == -1 or pos > last_pos:
            correct += 1                     # từ đầu tiên, hoặc đứng sau từ trước → đúng vị trí
            last_pos = pos
        else:
            misplaced += 1                   # xuất hiện trước từ lẽ ra đứng trước → sai vị trí

    total = len(target_seq)
    if total == 0:
        return 100.0

    # đúng ăn full, sai vị trí ăn nửa điểm, thiếu = 0
    return (correct * 1.0 + misplaced * 0.5) / total * 100

Logic cộng điểm: đúng vị trí ×1.0, sai vị trí ×0.5 (vì dù sao cũng nói được từ đó), thiếu ×0. Mẫu số là tổng số từ target → từ thiếu tự động bị phạt qua mẫu số. Cách này phạt "thiếu từ" nặng hơn "sai vị trí", đúng với trực giác lâm sàng.

Fluency Component

Dùng chung cho NAM, CMD Mode 2, và SEN. So sánh thời lượng audio bệnh nhân với thời lượng chuẩn của TTS pre-generated cho bài đó.

// fluency_score formula
ratio = min(actual_dur, expected_dur) / max(actual_dur, expected_dur)
fluency_score = ratio × 100  if  ratio ≥ 0.5  else  0
actual_dur
Thời lượng audio bệnh nhân (giây), từ AudioInput module. Tối đa 10s.
expected_dur
Field duration_expected trong Exercise schema, lấy từ TTS pre-generated.
ngưỡng 0.5
Nếu ratio < 0.5 → bệnh nhân nói quá nhanh hoặc quá chậm → fluency = 0, không phạt toàn bộ score.
Fluency chỉ chiếm 10–20% tổng score. Bệnh nhân nói đúng nhưng chậm sẽ bị giảm fluency nhẹ, không bị fail toàn bài. Đây là thiết kế có chủ ý — aphasia thường gây chậm, không nên phạt nặng yếu tố này.
python
def fluency_score(actual_dur: float, expected_dur: float) -> float:
    if expected_dur <= 0 or actual_dur <= 0:
        return 70.0  # fallback nếu thiếu data
    ratio = min(actual_dur, expected_dur) / max(actual_dur, expected_dur)
    return ratio * 100 if ratio >= 0.5 else 0.0

// 08

Result Classification — Pass / Near / Retry

Sau khi tính score, Rule Engine phân loại kết quả và quyết định hành động tiếp theo. Ngưỡng phân loại giống nhau cho tất cả exercise type (trừ CMD Mode 1 — xem bên dưới).

PASS
≥ 70
Chuyển bài tiếp theo. Cộng vào correct_streak. Phát feedback tích cực.
NEAR
50 – 69
Phát TTS chậm 0.75×. Cho phép thử lại (tối đa 3 lần). Reset correct_streak.
RETRY
< 50
Phát TTS 0.6×. Nếu đã 3 lần → skip + log. Reset correct_streak.

// 09

Progression Rule & Level-up Logic

Vocab level là cơ chế tăng độ khó từ vựng dần trong một buổi luyện tập. Nó không phải trạng thái lưu lâu dài của bệnh nhân.

Vocab level chỉ tồn tại trong session: Mỗi session bắt đầu ở Level 1. Trong session, cứ 3 bài đúng liên tiếp cùng topic thì vocab level tăng 1 (có thể lên nhiều lần trong cùng session). Hết session (làm xong 10 bài) → vocab level reset về 1 cho buổi sau. Đây là adaptive difficulty trong buổi, không phải xếp hạng năng lực dài hạn.

Định nghĩa "đúng" — gộp chung mọi exercise type

Level up tính trên cùng topic, không phân biệt exercise type. Một bài được coi là "đúng" khi:

Exercise typeĐiều kiện "đúng" (tính vào streak)
NAMscore ≥ 70 (PASS)
CMD Mode 1 (Recognition)result == Correct
CMD Mode 2 (Repetition)score ≥ 70 (PASS)
SENscore ≥ 70 (PASS)

    correct_streak: int   # số bài đúng liên tiếp cùng topic
) -> ProgressionUpdate:

    if is_correct(result, exercise_type):
        correct_streak += 1
    else:
        correct_streak = 0      # sai/incorrect/near → reset chuỗi

    # 3 bài đúng liên tiếp → lên 1 level (tối đa level 3)
    if correct_streak >= 3 and vocab_level < 3:
        return ProgressionUpdate(
            vocab_level    = vocab_level + 1,
            correct_streak = 0,           # reset streak sau khi lên level
            action         = "level_up"
        )

    return ProgressionUpdate(
        vocab_level    = vocab_level,
        correct_streak = correct_streak,
        action         = "hold"
    )

def on_session_end():
    # Reset vocab_level và streak về mặc định cho session sau
    vocab_level    = 1
    correct_streak = 0

// 10

Vocab Level Difficulty Weighting

Vocab Level (1–2–3) — chính là level từ vựng tại thời điểm làm bài trong session (cơ chế tăng dần độ khó ở Section 09) — được dùng để tính difficulty_weighted_score, một chỉ số riêng phục vụ SLP Dashboard. Nó không ảnh hưởng đến raw_score và không ảnh hưởng đến progression.

Nguyên tắc tách biệt: raw_score (0–100) dùng cho progression rule và feedback UI. difficulty_weighted_score chỉ dùng trong SLP dashboard để thể hiện tiến bộ thực chất — một bài đúng ở level 3 (từ khó hơn) có giá trị lâm sàng cao hơn cùng điểm ở level 1. Vì level tăng dần trong session, các bài về sau của buổi (nếu bệnh nhân làm tốt) sẽ có weighted_score cao hơn.

// difficulty_weighted_score
weighted = raw_score × difficulty_multiplier[vocab_level]
Level 1 — Easy
multiplier = 1.00. Từ đơn, danh từ quen thuộc (sách lớp 1–3).
Level 2 — Medium
multiplier = 1.15. Từ ghép đời thường, động từ cơ bản.
Level 3 — Hard
multiplier = 1.30. Câu giao tiếp, mệnh đề phức.
weighted_score có thể vượt 100. SLP dashboard hiển thị theo thang gốc (vd: "97.5 / 130 max ở level 3") hoặc normalize về 0–100 tùy UI.
python
DIFFICULTY_MULTIPLIER = {1: 1.00, 2: 1.15, 3: 1.30}

def apply_difficulty_weight(raw_score: float, vocab_level: int) -> float:
    multiplier = DIFFICULTY_MULTIPLIER.get(vocab_level, 1.0)
    return round(raw_score * multiplier, 2)

// 11

Attempt & Fallback Handling (SEN)

Sentence Building có cơ chế fallback riêng: nếu bệnh nhân không đúng, hệ thống phát audio câu hoàn chỉnh để bệnh nhân nghe và lặp lại. Score của attempt 2 (sau fallback) bị giảm bằng multiplier 0.75 để phân biệt với attempt tự làm.

// sentence_building — adjusted score per attempt
adjusted_score = raw_score × attempt_multiplier[attempt_number]
attempt 1
multiplier = 1.00. Bệnh nhân chưa nghe fallback audio. Score nguyên.
attempt 2
multiplier = 0.75. Đã nghe full sentence audio. Score giảm 25%.
attempt 3
multiplier = 0.60. Đã nghe 2 lần. Score giảm 40%. Sau đó skip.
adjusted_score là score được log vào database và dùng cho progression rule. Raw score (attempt 1) được lưu riêng để SLP xem nguyên bản.
pythonscoring_engine/sentence_builder.py
ATTEMPT_MULTIPLIER = {1: 1.00, 2: 0.75, 3: 0.60}

def score_sentence(
    exercise: Exercise,
    transcript: str,
    audio_duration: float,
    attempt_number: int
) -> ScoreResult:

    raw = (
        0.40 * keyword_score(transcript, exercise.missing_words) +
        0.50 * order_score(transcript, exercise.full_sentence) +
        0.10 * fluency_score(audio_duration, exercise.duration_expected)
    )

    multiplier = ATTEMPT_MULTIPLIER.get(attempt_number, 0.60)
    adjusted   = raw * multiplier

    return ScoreResult(
        score         = adjusted,
        raw_score     = raw,
        attempt       = attempt_number,
        used_fallback = attempt_number > 1,
        result        = classify(adjusted),
        components    = {
            "keyword":      ...,
            "order_score":  ...,
            "fluency":      ...,
        }
    )
